# CryptoSafe Manager - техническое резюме

## Архитектура

Проект разделен на три основных слоя:

- `src/gui`: графический интерфейс на Tkinter, диалоги, таблицы, настройки, системный трей и пользовательские уведомления.
- `src/core`: бизнес-логика приложения, криптография, операции с хранилищем, буфер обмена, аудит, импорт/экспорт и защитные механизмы.
- `src/database`: работа с SQLite, миграции схемы, индексы, транзакции, резервное копирование и восстановление.

Компоненты связываются через `EventBus`. Такой подход снижает жесткую связанность: журнал аудита, буфер обмена, режим паники и графический интерфейс могут реагировать на события друг друга без прямых зависимостей между всеми модулями.

## Криптографические решения

Основная цель криптографического слоя - хранить пользовательские записи только в зашифрованном виде и не использовать мастер-пароль напрямую как ключ шифрования.

- **Проверка мастер-пароля.** Для проверки пароля используются функции получения ключа из `src/core/crypto/key_derivation.py`. В проекте применяются параметры Argon2id/PBKDF2, соль и отдельные данные проверки, чтобы один и тот же пароль не давал одинаковые проверочные значения в разных хранилищах.
- **Получение ключей.** Мастер-пароль проходит через функцию получения производного ключа: на вход подаются пароль, соль и параметры сложности, на выходе получается криптографический материал для дальнейшей работы. Это замедляет перебор паролей и делает атаку по украденной базе дороже.
- **Разделение ключей.** Проверка пароля и шифрование данных разделены. Проверочное значение используется для аутентификации пользователя, а ключ шифрования применяется для работы с записями. Это снижает риск повторного использования одного секрета в разных задачах.
- **Шифрование записей.** Записи хранилища шифруются через AES-256-GCM. Этот режим одновременно обеспечивает конфиденциальность и проверку целостности: если зашифрованные данные или тег аутентичности изменить, расшифровка завершится ошибкой.
- **Уникальные nonce.** Для каждой операции шифрования используется уникальный nonce. Это критично для AES-GCM, потому что повторное использование nonce с одним ключом может ослабить защиту.
- **Защита экспорта.** Экспорт в защищенном формате также шифруется через AES-GCM. Ключ для экспортного файла получается из пароля экспорта, а метаданные хранят только технические параметры, необходимые для последующей расшифровки, без открытых секретов.
- **Подпись и аудит.** Журнал аудита поддерживает подписи и проверку целостности цепочки событий. Это помогает обнаружить удаление, подмену или нарушение порядка записей аудита.
- **Безопасное сравнение.** Сравнение критичных значений выполняется через функции сравнения за постоянное время, чтобы уменьшить риск утечки информации по времени выполнения.

Криптографический код покрывается модульными тестами: проверяется шифрование и расшифровка, уникальность nonce, реакция на повреждение тега аутентичности, согласованность получения ключа и сценарии экспорта/импорта.

## Защита памяти и побочных каналов

Механизмы усиления безопасности находятся в `src/core/security`. Они не заменяют криптографию, а закрывают дополнительные риски: остатки секретов в памяти, утечки по времени выполнения, неактивную сессию и экстренные сценарии.

### Защита памяти

`src/core/security/memory_guard.py` отвечает за контролируемую работу с чувствительными буферами.

- **Отслеживаемые выделения.** Чувствительные данные помещаются в управляемые буферы, которые можно централизованно очистить при блокировке, выходе или режиме паники.
- **Затирание памяти.** Для изменяемых буферов используется перезапись нулями. Это уменьшает вероятность, что пароль, ключ или временный секрет останется в памяти после использования.
- **Контекст чувствительных данных.** `SensitiveScope` позволяет регистрировать временные буферы и автоматически очищать их при выходе из блока выполнения.
- **Канареечные значения.** Вокруг защищенных областей используются контрольные значения. Если соседняя память повреждена, проверка канареек позволяет обнаружить нарушение.
- **Глобальная аварийная очистка.** При режиме паники вызывается общий wipe для зарегистрированных секретов и кэшей ключей.

Полностью гарантировать отсутствие секретов в памяти в Python невозможно из-за сборщика мусора, immutable-строк и внутренних копий интерпретатора. Поэтому реализация использует практический подход: секреты по возможности хранятся в изменяемых буферах, очищаются сразу после использования и дополнительно стираются при блокировке приложения.

### Защита от побочных каналов

`src/core/security/side_channel_protection.py` снижает риск атак, основанных на наблюдении времени выполнения.

- **Сравнение за постоянное время.** Для паролей, токенов, хэшей и проверочных значений используется сравнение, время которого меньше зависит от позиции первого отличающегося символа.
- **Безопасная проверка пароля.** Обертки проверки пароля используют единый путь выполнения для успешных и неуспешных проверок, чтобы уменьшить различия в поведении.
- **Стабилизация операций поиска.** Для чувствительных проверок применяется подход, который сохраняет обычную семантику поиска, но снижает выраженность различий по времени.
- **Настраиваемый криптографический jitter.** Для отдельных операций может добавляться небольшая случайная задержка. Она усложняет точные измерения времени, но остается ограниченной, чтобы не портить отзывчивость интерфейса.

Эти меры особенно важны для сценариев проверки пароля, сравнения секретов и обработки защищенного кэша. В тестах проверяются корректность сравнения за постоянное время, измеримые накладные расходы и ограниченность временного разброса.

### Автоблокировка и режим паники

- `activity_monitor.py` отслеживает активность пользователя, фокус окна и таймаут бездействия. При превышении таймаута приложение блокирует хранилище.
- `panic_mode.py` выполняет экстренные действия: блокирует хранилище, очищает буфер обмена, стирает чувствительные буферы и может скрыть или закрыть приложение в зависимости от настроек.
- `platform_security.py` определяет возможности операционной системы и выбирает безопасные запасные сценарии, если конкретная системная функция недоступна.
- `security_validator.py` формирует отчеты по требованиям безопасности и производительности, чтобы можно было показать соответствие спринту.

## База данных

SQLite используется как локальное хранилище. Основные таблицы:

- `vault_entries`: зашифрованные записи хранилища.
- `deleted_entries`: мягкое удаление и восстановление записей.
- `settings`: настройки приложения.
- `key_store`: параметры и данные управления ключами.
- `audit_log`, `audit_keys`, `audit_security_events`: подписанный аудит и события безопасности.
- `import_export_history`: история операций импорта и экспорта.
- `shared_entries`, `contacts`: безопасный обмен записями и контакты для передачи ключей.

Миграции выполняются в `DatabaseHelper` при инициализации. Версия схемы хранится через `PRAGMA user_version`. Операции записи выполняются через транзакции, чтобы при ошибках не оставлять базу в частично измененном состоянии.

## Импорт и экспорт

`src/core/import_export` реализует:

- собственный зашифрованный JSON-формат;
- импорт и экспорт CSV с очисткой опасных значений;
- совместимость с форматами Bitwarden и LastPass;
- предварительную проверку импорта без записи в базу;
- режимы слияния, замены и восстановления через checkpoints;
- защищенный обмен через пароль, RSA/ECC и QR-разбиение.

Режим паники может прервать операции импорта и экспорта через шину событий. Это нужно, чтобы при экстренной блокировке не продолжалась долгая операция с чувствительными данными.

## Тестирование и покрытие

Тесты находятся в `tests/` и покрывают криптографию, CRUD-операции хранилища, буфер обмена, аудит, импорт/экспорт, режим паники, защиту памяти, защиту от побочных каналов и требования производительности.

Политика покрытия для Sprint 8 задана в `.coveragerc`. Графический интерфейс исключен из автоматического покрытия и проверяется ручными базовыми сценариями, потому что Tkinter-окна сложнее стабильно тестировать без графического окружения.

Команда финального отчета:

```bash
pytest tests/ --cov=src --cov-report=term --cov-report=html:tests/report/html
```

## Сборка

Сборка выполняется через PyInstaller:

```bash
python build.py
```

Результат сборки находится в `dist/CryptoSafeManager/`. Для передачи пользователям можно дополнительно использовать архив `dist/CryptoSafeManager.zip`, если он был создан через `build.py`.
